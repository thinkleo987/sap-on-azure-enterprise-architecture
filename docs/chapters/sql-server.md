# SAP on SQL Server for Azure Architecture

---

## Overview

Microsoft SQL Server is a fully supported SAP DBMS platform that serves as the database engine for SAP NetWeaver ABAP, SAP S/4HANA (prior to the mandatory HANA migration path), SAP Business Warehouse, SAP Solution Manager, SAP Central Services infrastructure databases, and the full landscape of SAP NetWeaver-based products. Within SAP landscapes, SQL Server provides the relational persistence layer for SAP ABAP-based systems running on Windows Server, managing table data, index structures, transaction logs, and the SAP ABAP data dictionary objects. In Azure environments, SQL Server for SAP is deployed as SQL Server on Azure Virtual Machines — not Azure SQL Database or Azure SQL Managed Instance — because the SAP NetWeaver kernel communicates directly with the SQL Server engine via the SAP DBSL (Database Shared Library) using specific internal APIs that are only compatible with the full SQL Server engine installation on a local operating system. SAP has certified SQL Server 2019 and SQL Server 2022 for use with SAP NetWeaver, SAP S/4HANA on SQL Server (for the transition period before HANA migration), SAP BW on SQL Server, and SAP Solution Manager 7.2.

Azure provides a native ecosystem for SQL Server on Virtual Machines that aligns directly with the SAP DBMS requirements. SQL Server on Azure VMs is covered under SAP Note 1928533 (SAP Applications on Azure: Supported Products and Azure VM Types), which maintains a current list of VM families and sizes certified for SAP DBMS workloads running SQL Server. Azure M-series and E-series virtual machines are the primary target for SAP SQL Server production deployments, with M-series preferred for memory-intensive SAP BW workloads and E-series for standard SAP ECC and S/4HANA-on-SQLServer landscapes. SQL Server on Azure VMs benefits from Azure-managed infrastructure including Azure Premium SSD v2 managed disks, Azure Ultra Disk for log volumes requiring sub-millisecond latency, Azure Backup integration via SQL Server IaaS Agent Extension for automated database backups to Recovery Services Vaults, and SQL Server Always On Availability Groups for high availability and disaster recovery. Microsoft certifies specific SQL Server versions and editions for SAP workloads and maintains published guidance for storage layout, OS configuration, and network settings through the Azure Architecture Center and SAP on Azure documentation set.

Key architectural decisions for SAP on SQL Server on Azure centre on three interdependent concerns: VM family selection, storage architecture, and high availability method. VM family selection determines both the compute characteristics and the maximum IOPS achievable through managed disk attachments; E-series VMs (Edsv5) provide the best IOPS-per-core ratio for SQL Server data volumes while M-series provide the memory density needed for large SAP BW systems running column-store aggregates in SQL Server. Storage architecture determines the separation of SQL Server data files (MDF/NDF), transaction log files (LDF), TempDB, and backup streams across independent disk volumes configured with Premium SSD v2 or Ultra Disk — this separation is mandatory for SAP systems to avoid I/O contention between the OLTP write-ahead log stream and the random read pattern of the data pages. High availability via SQL Server Always On Availability Groups deployed across Availability Zones, fronted by an Azure Internal Load Balancer with a Distributed Network Name listener, is the architecturally correct approach for SAP production systems, replacing the deprecated Windows Server Failover Cluster with Shared Disk (WSFC+SCSI-3 PR) model that is not supported on Azure.
